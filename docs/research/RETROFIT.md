# SPECTRA Retrofit Protocol — Brownfield Installation Guide

How to install SPECTRA in an existing codebase so that specifications are indistinguishable from what a senior engineer on the team would produce.

**Run once per project.** The retrofit produces `spectra-conventions.md` — a convention map that the SPECTRA planning cycle consumes as structural context. After installation, the core cycle runs with zero retrofit overhead.

For greenfield projects, `spectra-init.sh` alone is sufficient. This document covers the deeper analysis needed for brownfield codebases with established patterns, existing convention files, and architectural decisions already in place.

---

## 1. Why Brownfield Is Different

Greenfield projects have no existing conventions to discover — the adaptation prompt generates conventions from framework defaults. Brownfield projects have:

- **Existing convention files** (`.cursorrules`, `CLAUDE.md`, `AGENTS.md`, etc.) that encode team decisions
- **Implicit conventions** visible only in the code (naming patterns, architectural boundaries, test strategies)
- **Structural complexity** (dependency graphs, module boundaries, hotspots) that affect planning

65% of developers report AI misses context during refactoring (Qodo, 2025 State of AI Code Quality, N=609). The retrofit protocol closes this gap by extracting conventions *before* the first planning session, not during it.

### Convention Files as an Ecosystem Pattern

| Tool | Convention Mechanism | Key Property |
|------|---------------------|-------------|
| **Cursor** | `.cursor/rules/*.mdc` with YAML frontmatter and globs | Scopeable (path-specific rules) |
| **Claude Code** | `CLAUDE.md` with `@imports` (5 levels) | Hierarchical (files reference other files) |
| **GitHub Copilot** | `.github/copilot-instructions.md` + path-specific `*.instructions.md` | Path-scoped |
| **AGENTS.md** | Linux Foundation / Agentic AI Foundation standard | Universal, agent-agnostic |

An empirical study of 401 repositories with cursor rules (Jiang & Nam, MSR 2026) identified 5 categories developers encode:

| Category | % of Repos | What It Captures |
|----------|-----------|-----------------|
| **Convention** | 77.6% | Code style, naming, project structure |
| **Guideline** | 77.3% | Architecture rules, testing requirements, deployment constraints |
| **Project Information** | 66.3% | Stack description, architecture overview, setup commands |
| **LLM Directive** | 60.1% | Response format, tool usage, behavioral constraints |
| **Example** | 48.1% | Code snippets, patterns, anti-patterns |

### Theoretical Basis

Convention files reduce plan entropy by eliminating decision alternatives. See [THEORY.md §10](THEORY.md#10-convention-files-as-entropy-reduction) for the formal treatment. A codebase with comprehensive conventions has lower plan entropy, requiring fewer verification layers — exactly what the adaptive verification budget prescribes.

This follows RepoGraph's proven pattern (Ouyang et al., ICLR 2025): structured repository context, provided as a modular plug-in, improves both procedural and agent-based frameworks by 32.8% avg without changing the framework itself. The retrofit protocol enriches context without changing SPECTRA's phases.

## 2. Convention File Taxonomy

Understanding what kinds of conventions exist determines what the retrofit needs to extract.

### By Authority

| Type | Description | Precedence |
|------|------------|-----------|
| **Prescriptive** | Rules the team has explicitly decided (naming, boundaries, test requirements) | Highest — override everything else |
| **Descriptive** | Documentation of how things currently work (architecture overviews, README) | Context for the adaptation prompt |
| **Structural** | Machine-readable project metadata (linter configs, CI definitions, manifests) | Validation source for gates |

### By Source

| Source | Files | Reliability |
|--------|-------|------------|
| **Human-authored** | `.cursorrules`, `CLAUDE.md`, `AGENTS.md`, `CONTRIBUTING.md`, `ARCHITECTURE.md`, ADRs | High — reflects team intent |
| **Tool-generated** | `.eslintrc`, `.rubocop.yml`, `.prettierrc`, `pyproject.toml` [tool.ruff] | High — enforced in CI |
| **Inferred** | `spectra-conventions.md` (generated by LLM from code samples) | Medium — requires cross-validation |

**Precedence rule:** Human-authored > Tool-generated > Inferred. When conventions conflict, explicit team decisions always win.

## 3. Installation Depth

Not all brownfield projects need the same depth of analysis. Choose based on codebase complexity. The EVPI principle applies ([THEORY.md §3](THEORY.md#3-confidence-gating-as-expected-value-of-information)): invest in deeper analysis only when the cost is less than the expected rework savings.

### Standard Install (minutes)

**When:** Most projects. Codebase has some convention files or clear framework conventions.

1. Run `spectra-init.sh` — detects stack, convention files, project structure
2. Review `spectra-project-profile.md` for accuracy
3. Paste `spectra-adaptation-prompt.md` into any LLM
4. Save output as `spectra-conventions.md`
5. Commit `spectra-conventions.md` to the repository

The adaptation prompt already includes detected convention files and their content summaries. The LLM merges these with framework-specific best practices.

### Deep Install (hours)

**When:** Large codebase (>500 files), no existing convention files, complex architecture, or multiple teams with inconsistent patterns.

Everything from Standard Install, plus:

1. **Identify exemplar files** — Find the "best" files that represent current team standards:
   - High-importance (many dependents)
   - Recently modified (reflects current, not legacy, standards)
   - Well-documented
   - Cover different architectural layers (not all models)
   - Select 5–8 using diversity: `relevance - 0.3 × redundancy`

2. **Feed exemplar files to the LLM** alongside the adaptation prompt, requesting extraction of:
   - Naming conventions (classes, methods, files, variables, constants)
   - Architectural conventions (business logic organization, data access, API structure, validation, error handling)
   - Testing conventions (framework, file location, naming, coverage)
   - Documentation conventions (docstring style, inline comments)
   - Workflow conventions (commit format, branch naming, PR process)

3. **Cross-validate** inferred conventions against:
   - Existing convention files (conflicts?)
   - Linter/formatter configs (`.rubocop.yml`, `.eslintrc`, `.prettierrc`)
   - CI pipeline definitions (what gates are enforced?)

4. **Merge and save** as `spectra-conventions.md` with confidence annotations (HIGH/MEDIUM/LOW per convention, with evidence).

### Structural Install (advanced, optional)

**When:** Monorepo, microservices, or any codebase where dependency structure is non-obvious.

Everything from Deep Install, plus structural analysis to identify:

- **Core abstractions**: High importance + low churn (stable foundations — handle with care in specs)
- **Active hotspots**: High importance + high churn (risky areas needing careful specs)
- **Module boundaries**: Clusters in the dependency graph with few cross-edges
- **Architectural patterns**: MVC, service layer, repository pattern, event-driven — detected from import structure, not directory names

This level typically requires tooling (tree-sitter AST parsing, dependency graph construction, PageRank ranking). Implementation details are outside this repository's scope per [CONTRIBUTING.md](../../.github/CONTRIBUTING.md). See §6 for guidance.

## 4. The Convention Map

The output of any installation depth is `spectra-conventions.md` — a structured mapping from SPECTRA's generic vocabulary to the target project's patterns.

See [templates.md](../spectra-methodology/templates.md) for the full Convention Map format. The key sections:

| Section | What It Contains | Used By |
|---------|-----------------|---------|
| **Quick Reference** | Top 5 conventions for this project | Every SPECTRA session (CLARIFY step 4) |
| **Convention Mapping** | SPECTRA concept → project class/path/exemplar | Pattern phase (step 2) |
| **Action Verb Mapping** | SPECTRA verb → project-specific phrasing | Construct phase (story action plans) |
| **Validation Gates** | P0/P1/P2 gates specific to this project | Test phase, Construct (agent hints) |
| **Architectural Boundaries** | Hard constraints (e.g., "no logic in controllers") | Explore (hypothesis constraints), Test (adversarial layer) |

**Storage:** `spectra-conventions.md` in the project root. Commit it — it's team knowledge.

**Freshness:** Re-run the retrofit when the project's architecture changes significantly (new framework, major refactor, team convention update). Incremental updates are fine — edit the file directly.

## 6. Reference Implementation Notes

The retrofit protocol describes *what* to extract and *how* to structure the output. The actual analysis can range from simple to sophisticated:

**Standard:** `spectra-init.sh` + any LLM. No additional tooling required. Covers most projects.

**Deep:** Manual exemplar selection + LLM extraction prompt. The prompt itself is a methodology artifact (see the adaptation prompt generated by `spectra-init.sh`).

**Structural:** Would include AST parsing (e.g., tree-sitter), dependency graph construction, importance ranking (e.g., PageRank). These are implementation choices, not methodology prescriptions.

**Implementation details — including specific tooling, language choices, and architectural decisions — are outside this repository's scope per [CONTRIBUTING.md](../../.github/CONTRIBUTING.md).** SPECTRA is a methodology; code belongs in your project or a companion tooling repository.
